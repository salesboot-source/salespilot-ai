# Requirements Document

## Introduction

SalesPilot AI requires a complete SaaS Subscription and Billing System to monetize the platform. The system supports four pricing tiers (Free, Starter, Professional, Business) with monthly and yearly billing cycles, integrates Xendit as the payment gateway for Indonesian payment methods, manages subscription lifecycles (upgrades, downgrades, cancellations, trials), enforces usage limits per plan, and provides both user-facing billing dashboards and admin revenue analytics.

## Glossary

- **Billing_System**: The overall subscription and billing module within SalesPilot AI responsible for plan management, payment processing, and subscription lifecycle operations
- **Plan_Catalog**: The component that stores and serves available pricing plans with their features, limits, and pricing
- **Subscription_Engine**: The component responsible for managing subscription state transitions including creation, upgrades, downgrades, cancellation, and renewal
- **Payment_Processor**: The component that interfaces with the Xendit API to create invoices, handle checkout redirections, and process payment results
- **Webhook_Handler**: The server-side component that receives, validates, and processes asynchronous payment event callbacks from Xendit
- **Usage_Tracker**: The component that monitors and enforces feature usage limits (prospect discovery count, export count, AI credits) per subscription plan
- **Billing_Dashboard**: The user-facing interface displaying subscription status, payment history, invoices, and usage metrics
- **Admin_Revenue_Dashboard**: The administrator interface displaying business metrics including MRR, ARR, churn rate, and subscription analytics
- **Trial_Manager**: The component that manages 7-day free trial lifecycle including activation, expiration tracking, and reminder notifications
- **Notification_Service**: The component responsible for sending transactional emails related to billing events (payment success, renewal reminders, trial expiring)
- **Xendit**: The third-party Indonesian payment gateway supporting Virtual Account, QRIS, Credit Card, E-Wallet, Retail Outlet, and Direct Debit payment methods
- **MRR**: Monthly Recurring Revenue — total predictable revenue normalized to a monthly amount
- **ARR**: Annual Recurring Revenue — MRR multiplied by 12
- **ARPU**: Average Revenue Per User — total revenue divided by active subscribers
- **Churn_Rate**: The percentage of subscribers who cancel within a given period
- **Proration**: Calculating a partial charge or credit when a user changes plans mid-billing cycle
- **Billing_Cycle**: The recurring period (monthly or yearly) for subscription charges
- **Invoice**: A payment record generated by Xendit containing amount, payment method, and status
- **Webhook**: An HTTP callback from Xendit to the Billing_System notifying of payment events

## Requirements

### Requirement 1: Plan Catalog Management

**User Story:** As a user, I want to view available subscription plans with clear feature comparisons, so that I can choose the plan that fits my needs.

#### Acceptance Criteria

1. THE Plan_Catalog SHALL store four plans: Free (Rp 0), Starter (Rp 299,000/month), Professional (Rp 799,000/month), and Business (Rp 1,999,000/month)
2. THE Plan_Catalog SHALL define feature limits for each plan including prospect discovery count, export capability, AI feature access, and support level
3. WHEN a user requests plan information, THE Plan_Catalog SHALL return both monthly and yearly pricing where yearly pricing applies a 20% discount
4. THE Plan_Catalog SHALL store the yearly prices as Starter (Rp 2,870,400/year), Professional (Rp 7,670,400/year), and Business (Rp 19,190,400/year)
5. THE Plan_Catalog SHALL mark the Professional plan as the recommended plan

### Requirement 2: Subscription Creation via Xendit Checkout

**User Story:** As a user, I want to select a plan and pay through familiar Indonesian payment methods, so that I can activate my subscription seamlessly.

#### Acceptance Criteria

1. WHEN a user selects a paid plan and billing cycle, THE Payment_Processor SHALL create a Xendit invoice with the correct amount and redirect the user to the Xendit checkout page
2. THE Payment_Processor SHALL support Virtual Account, QRIS, Credit Card, E-Wallet, Retail Outlet, and Direct Debit as payment methods via the Xendit invoice
3. WHEN the Xendit checkout session completes successfully, THE Payment_Processor SHALL redirect the user to a success page on the Billing_Dashboard
4. WHEN the Xendit checkout session fails or is abandoned, THE Payment_Processor SHALL redirect the user to a failure page with an option to retry
5. THE Payment_Processor SHALL include a unique idempotency key in each invoice creation request to prevent duplicate charges

### Requirement 3: Webhook Payment Verification

**User Story:** As a system operator, I want payment status verified exclusively through server-side webhooks, so that the system cannot be manipulated by frontend tampering.

#### Acceptance Criteria

1. WHEN an `invoice.paid` webhook is received, THE Webhook_Handler SHALL verify the webhook signature using the Xendit callback verification token
2. WHEN a verified `invoice.paid` event is received, THE Subscription_Engine SHALL activate the subscription, store the invoice record, generate a receipt, and trigger a confirmation email
3. WHEN an `invoice.expired` or `invoice.failed` webhook is received, THE Webhook_Handler SHALL mark the transaction as failed and notify the user
4. THE Webhook_Handler SHALL implement idempotent processing to prevent duplicate execution of the same webhook event
5. THE Webhook_Handler SHALL log every received webhook event with timestamp, event type, invoice ID, and processing result
6. THE Webhook_Handler SHALL respond with HTTP 200 within 5 seconds to acknowledge receipt of each webhook
7. IF a webhook signature verification fails, THEN THE Webhook_Handler SHALL reject the request with HTTP 401 and log the attempt

### Requirement 4: Subscription Lifecycle Management

**User Story:** As a subscriber, I want to upgrade, downgrade, or cancel my plan at any time, so that I have full control over my subscription.

#### Acceptance Criteria

1. WHEN a user upgrades to a higher plan, THE Subscription_Engine SHALL activate the new plan immediately and calculate prorated billing for the remaining days in the current cycle
2. WHEN a user downgrades to a lower plan, THE Subscription_Engine SHALL schedule the change to take effect at the end of the current billing period
3. WHEN a user cancels a subscription, THE Subscription_Engine SHALL maintain access until the current billing period expires and mark the subscription as pending cancellation
4. WHEN a cancelled subscription reaches its period end date, THE Subscription_Engine SHALL transition the user to the Free plan
5. WHEN a user with a pending cancellation requests to resume, THE Subscription_Engine SHALL remove the pending cancellation status and continue the subscription
6. THE Subscription_Engine SHALL record every state change in the subscription_logs table with the previous state, new state, timestamp, and trigger reason

### Requirement 5: Automatic Subscription Renewal

**User Story:** As a subscriber, I want my subscription to renew automatically, so that I maintain uninterrupted access to premium features.

#### Acceptance Criteria

1. WHEN a subscription billing period ends and the subscription is active, THE Subscription_Engine SHALL create a new Xendit invoice for the renewal amount
2. WHEN a renewal payment succeeds via webhook confirmation, THE Subscription_Engine SHALL extend the subscription period by one billing cycle
3. WHEN a renewal payment fails, THE Subscription_Engine SHALL send a payment failure notification and provide a 3-day grace period before downgrading to Free
4. THE Subscription_Engine SHALL send a renewal reminder email 3 days before the billing period ends

### Requirement 6: Free Trial Management

**User Story:** As a new user, I want to try Professional plan features for free before committing, so that I can evaluate the product before paying.

#### Acceptance Criteria

1. WHEN a new user registers, THE Trial_Manager SHALL offer a 7-day free trial with Professional plan features without requiring payment information
2. THE Trial_Manager SHALL track trial start date and expiration date per user
3. WHEN a trial has 3 days remaining, THE Trial_Manager SHALL send a trial expiring email and display a dashboard notification
4. WHEN a trial has 1 day remaining, THE Trial_Manager SHALL display an upgrade modal on the Billing_Dashboard
5. WHEN a trial expires without conversion, THE Trial_Manager SHALL transition the user to the Free plan
6. WHEN a user on trial subscribes to a paid plan, THE Trial_Manager SHALL end the trial immediately and activate the paid subscription

### Requirement 7: Usage Tracking and Enforcement

**User Story:** As a product owner, I want to enforce feature limits per plan, so that users are incentivized to upgrade for higher usage.

#### Acceptance Criteria

1. THE Usage_Tracker SHALL enforce the following prospect discovery limits: Free (20), Starter (200), Professional (2000), Business (unlimited)
2. THE Usage_Tracker SHALL reset usage counters at the start of each billing cycle
3. WHEN a user reaches a usage limit, THE Usage_Tracker SHALL block the action and display a message indicating the limit and an option to upgrade
4. THE Usage_Tracker SHALL track separate counters for prospect discovery, exports, and AI feature usage
5. WHEN a user queries usage status, THE Usage_Tracker SHALL return current usage, maximum allowed, and billing cycle reset date

### Requirement 8: User Billing Dashboard

**User Story:** As a subscriber, I want to view and manage my subscription details in one place, so that I have transparency over my billing.

#### Acceptance Criteria

1. THE Billing_Dashboard SHALL display the current plan name, subscription status, renewal date, next payment amount, and billing cycle
2. THE Billing_Dashboard SHALL display a usage meter showing prospect count used versus limit, export count, and AI credits remaining
3. THE Billing_Dashboard SHALL display a payment history list with date, amount, status, and payment method for each transaction
4. THE Billing_Dashboard SHALL provide actions to upgrade plan, downgrade plan, cancel subscription, resume subscription, download invoice as PDF, change billing cycle, and change payment method
5. WHEN a user clicks download invoice, THE Billing_Dashboard SHALL generate and serve a PDF receipt for the selected transaction

### Requirement 9: Admin Revenue Dashboard

**User Story:** As an admin, I want to monitor revenue metrics and subscription health, so that I can make informed business decisions.

#### Acceptance Criteria

1. THE Admin_Revenue_Dashboard SHALL display MRR, ARR, total revenue, and today's revenue as summary cards
2. THE Admin_Revenue_Dashboard SHALL display active subscriber count, trial user count, churn rate, conversion rate, and ARPU
3. THE Admin_Revenue_Dashboard SHALL display a revenue chart showing daily or monthly revenue over a selectable time range
4. THE Admin_Revenue_Dashboard SHALL display a subscription growth chart showing new subscriptions, cancellations, and net growth over time
5. THE Admin_Revenue_Dashboard SHALL display a plan distribution breakdown showing subscriber count per plan
6. THE Admin_Revenue_Dashboard SHALL display a list of the 20 most recent transactions with user, plan, amount, status, and timestamp
7. THE Admin_Revenue_Dashboard SHALL restrict access to users with admin role only

### Requirement 10: Billing Notification Emails

**User Story:** As a user, I want to receive timely email notifications about my billing activity, so that I stay informed about charges and subscription changes.

#### Acceptance Criteria

1. WHEN a user completes registration, THE Notification_Service SHALL send a welcome email
2. WHEN a trial is activated, THE Notification_Service SHALL send a trial started email specifying the trial end date
3. WHEN a payment succeeds, THE Notification_Service SHALL send a payment success email with amount, plan, and next renewal date
4. WHEN a subscription is about to renew within 3 days, THE Notification_Service SHALL send a renewal reminder email with the upcoming charge amount
5. WHEN a payment fails, THE Notification_Service SHALL send a payment failed email with instructions to update payment method
6. WHEN a subscription is cancelled, THE Notification_Service SHALL send a cancellation confirmation email stating the access end date
7. WHEN a plan upgrade or downgrade occurs, THE Notification_Service SHALL send a plan change confirmation email with effective date and new plan details
8. WHEN a trial is expiring within 3 days, THE Notification_Service SHALL send a trial expiring email with upgrade options

### Requirement 11: Billing Security

**User Story:** As a system operator, I want billing operations to be secure against manipulation, so that revenue integrity is maintained.

#### Acceptance Criteria

1. THE Billing_System SHALL verify subscription status exclusively through server-side webhook confirmations and database state, ignoring frontend-provided payment status
2. THE Billing_System SHALL protect all billing API endpoints with JWT authentication using the existing auth system
3. THE Webhook_Handler SHALL validate the Xendit callback verification token on every incoming webhook request before processing
4. THE Billing_System SHALL use database transactions for all operations that modify subscription state and financial records simultaneously
5. THE Billing_System SHALL store payment-related secrets (Xendit API key, webhook verification token) exclusively in server-side environment variables
6. THE Billing_System SHALL rate-limit the checkout creation endpoint to 5 requests per minute per user to prevent abuse

### Requirement 12: Billing Cycle Toggle

**User Story:** As a subscriber, I want to switch between monthly and yearly billing, so that I can optimize my spending with annual discounts.

#### Acceptance Criteria

1. WHEN a user switches from monthly to yearly billing, THE Subscription_Engine SHALL calculate the prorated credit for remaining monthly days and apply it toward the yearly price
2. WHEN a user switches from yearly to monthly billing, THE Subscription_Engine SHALL schedule the change to take effect at the end of the current yearly period
3. THE Billing_Dashboard SHALL display a toggle showing monthly and yearly pricing with the savings amount clearly indicated

### Requirement 13: Database Schema for Billing

**User Story:** As a developer, I want a well-structured database schema for billing data, so that the system can efficiently store and query subscription and payment records.

#### Acceptance Criteria

1. THE Billing_System SHALL create a `plans` table storing plan ID, name, monthly price, yearly price, feature limits as JSONB, and display order
2. THE Billing_System SHALL create a `subscriptions` table storing subscription ID, user ID, plan ID, status, billing cycle, current period start, current period end, trial end date, cancelled at timestamp, and created/updated timestamps
3. THE Billing_System SHALL create a `transactions` table storing transaction ID, user ID, subscription ID, Xendit invoice ID, amount, currency, status, payment method, and timestamps
4. THE Billing_System SHALL create an `invoices` table storing invoice ID, transaction ID, user ID, invoice number, amount, tax, total, PDF URL, and issued date
5. THE Billing_System SHALL create a `payment_methods` table storing method ID, user ID, type, provider reference, last four digits or identifier, and active status
6. THE Billing_System SHALL create a `subscription_logs` table storing log ID, subscription ID, previous state, new state, reason, metadata as JSONB, and timestamp
7. THE Billing_System SHALL add a `plan_id` column and `subscription_status` column to the existing users table to enable quick plan-level access checks

### Requirement 14: Pricing Page UI

**User Story:** As a visitor or free user, I want an attractive pricing page that clearly communicates plan differences, so that I can make an informed purchase decision.

#### Acceptance Criteria

1. THE Billing_System SHALL display a pricing page with card-based layout showing all four plans side by side on desktop and stacked on mobile
2. THE Billing_System SHALL include a monthly/yearly toggle at the top of the pricing page that updates all displayed prices in real time
3. THE Billing_System SHALL highlight the Professional plan as "Most Popular" with distinct visual emphasis
4. THE Billing_System SHALL display a feature comparison checklist within each plan card indicating included and excluded features
5. THE Billing_System SHALL display a call-to-action button on each plan card labeled "Get Started" for Free, "Subscribe" for paid plans, and "Current Plan" (disabled) for the user's active plan
6. THE Billing_System SHALL render the pricing page with dark mode support, rounded cards, subtle glass effect, and smooth transition animations consistent with the existing SalesPilot UI design system
